# BNotes MD 圓桌會議：錯誤記憶庫與不重犯機制

建立日期：2026-05-17  
適用範圍：BNotes 既有文章升級、新文章發布前檢查、批次更新後複盤  
主責：BNotes PGM、內容總編、咖啡專家、SEO/GEO、品牌守門人、技術發布  
核心原則：不追究誰錯，透明檢視錯誤如何被看見、如何被修正、如何不再重犯。

## 一、制度目的

BNotes 的文章升級不是單純把檔案改完，而是要讓團隊越改越準、越改越快、越改越接近一次到位。

因此，本文件建立「錯誤記憶庫」與「不重犯機制」。所有錯誤都必須被轉成可檢查規則，而不是停留在口頭提醒。

這套制度的目的有三個：

1. 讓錯誤公開、透明、可討論。
2. 讓每次返工都轉化成下一次的效率。
3. 讓團隊從被動修正，進步成主動預判。

## 二、圓桌會議共識

錯誤不是用來追責，而是用來建立組織能力。

MD 委員會與專家團隊的共識如下：

- 不說「誰做錯」，而說「哪個檢查點缺失」。
- 不說「下次注意」，而要寫成「下次如何檢查」。
- 不說「已修正」，而要確認「source / dist / live 是否同步生效」。
- 不把董事會反饋當成個案，而要轉成團隊的可複用規則。
- 同一錯誤若重複出現，代表 SOP 或 QC 還不夠清楚，不只是執行端疏忽。

## 三、錯誤記憶庫格式

每一筆錯誤都必須用以下格式記錄：

```text
【錯誤編號】
BNotes-ERR-YYYYMMDD-序號

【錯誤名稱】
用一句話說明問題。

【發生文章】
文章 slug / 文章標題。

【問題描述】
讀者實際看到或感受到的問題。

【根本原因】
是結構檢查缺失、視覺審美缺失、語氣判斷缺失、技術同步缺失，或 SOP 未定義。

【正確標準】
下次應該長什麼樣子。

【檢查方法】
如何在修後自己發現，不需要等董事會指出。

【不重犯規則】
下次同類文章必須遵守的硬規則。

【是否需要工具化】
否 / 是：若可用 validator、腳本、CSS 檢查、live check 自動檢出，必須列入。
```

## 四、本次 Arabica vs Robusta 修正產生的錯誤記憶

### BNotes-ERR-20260517-01：只驗收導讀卡存在，未驗收閱讀位置

【發生文章】  
`arabica-vs-robusta-complete-guide`

【問題描述】  
文章一進正文就出現導讀卡，讀者還沒被開場故事帶入，就先看到整理重點。形式上有導讀卡，但閱讀節奏不符合 BNotes 專家朋友帶路的標準。

【根本原因】  
結構 QC 只檢查 `module-conclusion module-guide` 是否存在，沒有檢查導讀卡是否位於「開場段落之後、第一個主 H2 之前」。

【正確標準】  
開場情境 / 問題意識 -> 單一導讀卡 -> 正文第一個主 H2。

【檢查方法】  
修後必須確認第一段正文出現在導讀卡前，且導讀卡在第一個主 H2 前。

【不重犯規則】  
不可只驗收「有導讀卡」。必須驗收「導讀卡前是否已有自然開場」。

【是否需要工具化】  
是。後續 validator 應加入導讀卡位置檢查。

### BNotes-ERR-20260517-02：導讀卡清單間距過大

【發生文章】  
`arabica-vs-robusta-complete-guide`

【問題描述】  
導讀卡下方三點雖然存在，但視覺間距過大，讀者看起來像普通正文清單，不像一張緊湊、精準的導讀卡。

【根本原因】  
視覺 QC 沒有檢查導讀卡內 `ul/li` 的專用間距，導致清單沿用正文清單節奏。

【正確標準】  
導讀卡三點必須緊湊、清楚、易掃讀，像一張完整讀卡，而不是散開的正文列表。

【檢查方法】  
修後必須檢查 `.module-conclusion ul` 與 `.module-conclusion li` 是否有專用間距，並以 live page 實際閱讀確認。

【不重犯規則】  
任何導讀卡都不得只檢查 3 點數量，必須檢查清單視覺密度。

【是否需要工具化】  
部分可工具化。CSS 可檢查，視覺密度仍需人工審美 QC。

### BNotes-ERR-20260517-03：世界華人定位被硬寫成不自然標題

【發生文章】  
`arabica-vs-robusta-complete-guide`

【問題描述】  
標題「回到華語市場」雖符合品牌方向，但和前文「台北咖啡館」語感不對稱，讀起來像策略語言，不像自然文章段落。

【根本原因】  
品牌定位 QC 把「服務華語讀者」直接寫成標題，而不是讓文章自然面向華語讀者。

【正確標準】  
文章可以服務華語世界，但標題應自然、具體、讀者導向。例如「你買的咖啡豆裡有多少 Robusta？」比「回到華語市場」更自然。

【檢查方法】  
修後必須問：這個標題是讀者會想看的問題，還是內部策略語言？

【不重犯規則】  
不要把品牌定位硬塞成段落標題。台灣、華語世界、海外華人都可以出現，但必須自然服務內容。

【是否需要工具化】  
否。此項屬於內容總編與品牌守門人的語氣 QC。

### BNotes-ERR-20260517-04：FAQ 有內容但未真正折疊互動

【發生文章】  
`arabica-vs-robusta-complete-guide`

【問題描述】  
FAQ 內容存在，但讀者端沒有完成「點問題才展開答案」的互動期待。

【根本原因】  
技術 QC 只檢查 FAQ 5 題存在，沒有檢查 `.faq-answer` 是否預設隱藏、`.faq-item.open` 是否顯示、點擊是否切換 open 狀態。

【正確標準】  
FAQ 必須預設收合，點擊問題後答案展開，再點擊可收合。

【檢查方法】  
本地與 live page 均需檢查：

- `.faq-answer{display:none}`
- `.faq-item.open .faq-answer{display:block}`
- 有 FAQ accordion JS
- 實際點擊可展開

【不重犯規則】  
不可只驗收 FAQ 題數。必須驗收 FAQ 互動。

【是否需要工具化】  
是。後續 validator 應加入 FAQ CSS / JS / live interaction 檢查。

### BNotes-ERR-20260517-05：結構 QC 通過但閱讀審美 QC 未完成

【發生文章】  
`arabica-vs-robusta-complete-guide`

【問題描述】  
文章曾通過結構驗收，但仍被董事會指出間距、語氣、互動等問題。代表團隊當時只完成「元素存在檢查」，沒有完成「讀者實際感受檢查」。

【根本原因】  
QC 層級不足。缺少視覺密度、語意對稱、互動完成度三個檢查面。

【正確標準】  
每篇文章修後必須通過三層 QC：

1. 結構 QC：模組、順序、數量、source/dist/live。
2. 視覺 QC：間距、密度、手機閱讀、分享區版位。
3. 語氣 QC：標題自然度、BNotes 專家朋友語氣、世界華人視角是否自然。

【檢查方法】  
PGM 必須在回報時明確列出三層 QC 結果，不得只說「驗收通過」。

【不重犯規則】  
任何文章不得只用機器檢查結束。必須有人工閱讀審美檢查。

【是否需要工具化】  
部分可工具化。結構與部分 CSS 可工具化；審美與語氣必須由 MD / 總編 / 品牌守門人共同檢查。

### BNotes-ERR-20260517-06：互動腳本不可插入到 DOM 尚未載入的位置

【發生文章】  
`coffee-equipment-cleaning-bible`

【問題描述】  
批次更新 FAQ 折疊功能時，曾在本機修正階段發現 FAQ accordion script 被插入到 head 區塊過早的位置。雖然最後已在部署前修正，但這代表批次更新時不能只檢查「有 script」，還要檢查 script 是否位於頁尾或 DOMContentLoaded 後能作用的位置。

【根本原因】  
技術 QC 檢查了 `bnotes-faq-accordion` 字串與 `classList.toggle('open')`，但尚未把「插入位置」列為硬性檢查。

【正確標準】  
FAQ accordion script 應位於頁尾、FAQ HTML 之後，或明確包在 DOMContentLoaded 之後執行。不得插在 head 中尚未有 `.faq-question` 節點的位置。

【檢查方法】  
修後必須確認 `faq-section` 出現在 `bnotes-faq-accordion` 之前，並確認頁面中沒有提前關閉 `</body>` / `</html>` 的殘留錯位。

【不重犯規則】  
不可只驗收 FAQ JS 存在。必須驗收 FAQ HTML 在前、FAQ JS 在後、頁面結構完整。

【是否需要工具化】  
是。批次 validator 應加入 `faq-section index < bnotes-faq-accordion index` 與單一 `</body></html>` 結尾檢查。

### BNotes-ERR-20260517-07：舊 Hero 被隱藏後，新 Header 必須重新顯示主標

【發生文章】  
`coffee-equipment-cleaning-bible`

【問題描述】  
舊版 `.article-hero` 被統一外殼隱藏後，原本的 H1 也被隱藏；但新注入的 `bnotes-unified-header` 只留下移除重複 H1 的註解，沒有重新輸出可見主標題，導致讀者進入文章後看不到文章主標。

【根本原因】  
品牌外殼 QC 只檢查 Logo / 導讀卡 / FAQ / 分享區，沒有把「可見文章主標」列為硬性檢查。技術上則是 header injection 少了 `bnotes-title`。

【正確標準】  
若舊 hero 或舊 H1 被隱藏，新 header 必須顯示可見的 `.bnotes-title`，且內容應等於文章主標。

【檢查方法】  
修後必須確認 live page 中有 `.bnotes-title`，且不只是 meta title / hidden H1 存在。檢查重點是讀者肉眼可見的主標。

【不重犯規則】  
不可只驗收 metadata 或 hidden H1。任何文章外殼更新都必須驗收可見主標、Logo、副標、日期/作者/閱讀時間完整顯示。

【是否需要工具化】  
是。批次 validator 應加入 `.bnotes-title` 字串與 live DOM 可見性檢查。

### BNotes-ERR-20260517-08：分享按鈕存在，但分享模組樣式未完成

【錯誤現象】  
「你的磨豆機，上次清潔是什麼時候？」文章已有 `分享這篇` 文字與五個分享平台，但沒有同步標準 `.share-section` / `.share-btn` / 平台色 CSS，導致實際畫面不像 BNotes 統一分享模組。

【根本原因】  
QC 只檢查分享按鈕數量與位置，沒有檢查分享區的品牌樣式是否完整，也沒有把「視覺版位」納入同一層驗收。

【影響】  
讀者會感覺文章尾段的功能區與其他黃金標竿文章不同，削弱 BNotes 文章外殼一致性。

【不重犯規則】  
分享區 QC 必須同時檢查五件事：`分享這篇` 標籤、LINE/Facebook/Threads/X/複製連結五個平台、`.share-section` 版位、`.share-btn` 圓形 icon 樣式、平台色 CSS。只檢查按鈕存在不算通過。

【是否需要工具化】  
是。validator 必須加入 share CSS token 檢查，至少包含 `.share-section`、`.share-btn`、`.share-btn-line`、`.share-btn-copy`。

### BNotes-ERR-20260517-09：延伸閱讀有三篇，但學習旅程卡版型未同步

【錯誤現象】  
「你的磨豆機，上次清潔是什麼時候？」文章的延伸閱讀已有三篇推薦與學習方向小標，但 CSS 未完整對齊 `water-quality` / `processing-methods` 標竿版型，造成標題、卡片密度、hover、字級與品牌金色提示不一致。

【根本原因】  
QC 只檢查「延伸閱讀是否存在」與「是否有三篇」，沒有檢查延伸閱讀是否已成為 BNotes 統一的學習旅程卡。

【影響】  
讀者雖然看得到推薦文章，但尾段不像同一個 BNotes 知識系統，會降低品牌一致性與自修路徑的專業感。

【不重犯規則】  
延伸閱讀 QC 必須同時檢查六件事：`related-title` 金色左線標題、`related-grid` 三欄響應式密度、`related-card` 邊框與圓角、hover 上浮與金色邊框、`related-kicker` 學習方向、`related-card-title` 字級與行距。只檢查三篇連結存在不算通過。

【是否需要工具化】  
是。validator 必須加入 related CSS token 檢查，至少包含 `.related-title`、`.related-grid`、`.related-card:hover`、`.related-kicker`、`.related-card-title`。

### BNotes-ERR-20260518-10：文章標籤殘留在內頁，破壞品牌閱讀外殼

【錯誤現象】  
`taipei-coffee-map-2026` 等文章內頁曾出現 `article-tags` 或「標籤」模組。這種分類資訊可存在於後台、索引或內容治理層，但不應在標準文章內頁打斷讀者閱讀。

【根本原因】  
QC 過去只檢查 Article Shell 主要模組是否存在，沒有把「舊標籤模組不可見」列為硬性檢查。

【正確標準】  
BNotes 文章內頁應以主標、導讀卡、正文、FAQ、參考資料、分享、延伸閱讀形成一致閱讀路線；不顯示文章標籤區塊。

【不重犯規則】  
任何文章升級或發布前，PGM 必須確認 source / dist / live 均無 `article-tags`、`標籤：` 或類似舊式標籤模組。

【是否需要工具化】  
是。`validate-article.js` 已新增「文章頁不得殘留標籤區塊」BLOCK 檢查。

### BNotes-ERR-20260518-11：未來賽事或年度報告被誤寫成已發生文章

【錯誤現象】  
`wbc-2026-champion-technique-breakdown` 與 `taipei-coffee-expo-2026-annual-report` 屬於未來新聞 / 未來年度報告，不應在事件尚未發生前以確定語氣發布。

【根本原因】  
選文 QC 沒有在開工前先判斷「事件是否已發生」「冠軍是否已產生」「年度報告是否有資料基礎」，導致內容可能變成臆測。

【正確標準】  
未來事件只能寫成展前攻略、觀賽指南、學習路線、歷屆趨勢整理或背景知識，不可寫成冠軍拆解、年度總結或已發生報導。

【不重犯規則】  
凡 slug、標題或內容含 `champion`、`annual-report`、`2026`、冠軍、年報、回顧、完整報告等訊號，PGM 必須先做時間軸與來源驗證。若事件尚未發生，必須改定位、合併、延後或下架。

【是否需要工具化】  
部分需要。`validate-article.js` 已新增「發布日期不可晚於今天」BLOCK 檢查；事件真實性與語氣仍需內容總編、咖啡專家與 SEO/GEO 共同判斷。

### BNotes-ERR-20260518-12：相似文章未合併，造成知識庫重複與 SEO 分散

【錯誤現象】  
`taipei-specialty-cafes` 與 `taipei-coffee-map-2026` 主題高度相似，若兩篇同時存在，會讓讀者入口分散，也可能造成搜尋意圖重複。

【根本原因】  
早期文章更新以單篇為單位，缺少「同主題文章是否已存在」與「是否應合併成更強主頁」的 cluster ownership 檢查。

【正確標準】  
同一搜尋意圖應優先做成一篇更強的主頁；較弱或重複文章應合併、封存或 301 到最相關頁面。

【不重犯規則】  
每篇文章開工前，PGM 必須檢查同 cluster 是否已有主頁。若有高度相似文章，先決定主頁、支援頁、合併或 redirect，不得直接重修兩篇互搶流量的文章。

【是否需要工具化】  
部分需要。可由內容盤點表與內鏈檢查輔助，但是否合併仍需 PGM + SEO/GEO 判斷。

### BNotes-ERR-20260518-13：只補 metadata 或 hidden H1，不等於讀者看得到文章主標

【錯誤現象】  
`taipei-coffee-expo-2026-complete-guide`、`bala-2026-world-latte-art-champion-taiwan` 等文章曾出現讀者端缺少可見文章主標的問題。

【根本原因】  
QC 曾把 `<title>`、schema headline 或 hidden H1 當成文章有主標，但沒有從讀者視角檢查頁面上方是否真的顯示標準 BNotes 主標。

【正確標準】  
文章必須有可見的 `.bnotes-title` 或等效標準主標，並與文章實際定位一致。

【不重犯規則】  
任何文章外殼更新後，不能只檢查 HTML metadata，必須檢查讀者可見主標、Logo、副標、日期/作者/閱讀時間與內文開場是否連續。

【是否需要工具化】  
是。`validate-article.js` 已新增「可見 BNotes 文章主標」BLOCK 檢查。

### BNotes-ERR-20260518-14：真實性確認不足，尤其是冠軍、比賽、年份與人物文章

【錯誤現象】  
`bala-2026-world-latte-art-champion-taiwan` 需要確認人物、年份、賽事與名次是否真實。此類文章若來源不足，會直接傷害 BNotes 的專業可信度。

【根本原因】  
過去 QC 把 references 存在視為足夠，但沒有針對高風險事實類文章要求官方來源或可信新聞來源交叉確認。

【正確標準】  
涉及冠軍、世界第一、比賽結果、人物名次、價格、安全、健康等高風險主張時，至少需官方來源或可信媒體來源支撐，且文章語氣不得超出來源可證明的範圍。

【不重犯規則】  
高風險事實文章不得只靠一般描述或二手轉述。PGM 回報時必須明列「已確認來源」與「仍需保守表述的地方」。

【是否需要工具化】  
部分需要。來源區塊可由 validator 檢查存在；來源品質與主張是否過度，仍由咖啡專家、內容總編與品牌守門人共同判斷。

## 五、更新前自我檢查卡

每篇文章開工前，PGM 必須先回答：

1. 這篇的標竿文章是哪一篇？
2. 這篇是入門、比較、教學、診斷、文化、科學、選購，還是進階文章？
3. 開場應該先用故事、痛點、問題、場景，還是直接說明？
4. 導讀卡應該放在第幾段後？
5. 快速判斷是否需要？若需要，是否應放在主文脈絡後，而不是文章入口？
6. 這篇是否有地區語氣風險？台灣是否只是案例，而不是邊界？
7. FAQ 是否要對標 `processing-methods` / `water-quality` 的折疊式標準？
8. 分享這篇是否固定在參考資料後、延伸閱讀前，且具備標準 share CSS 與五個平台 icon？
9. 延伸閱讀是否有三篇聚焦推薦，且具備標準 related CSS 與學習方向小標？
10. 是否殘留文章標籤、舊版 share-bar、舊版 article-references、商業連結或內部碼？
11. 這篇是否屬於未來新聞、未來年度報告、未發生賽事冠軍拆解，或需要先改定位 / 合併 / 下架？
12. 是否有同主題相似文章需要合併、封存或 301 導流？
13. 若涉及冠軍、人物、年份、比賽、安全或研究主張，是否已用官方或可信來源確認？
14. 這篇完成後，讀者是否會感覺自己在 BNotes 的知識旅程裡？

## 七、錯誤進入硬規則的條件

以下情況必須把錯誤升級為硬規則：

1. 同一問題出現兩次以上。
2. 董事會明確指出問題。
3. 問題會影響全站一致性。
4. 問題會讓讀者感覺文章不像 BNotes。
5. 問題可被工具檢查，但目前尚未納入 validator。

硬規則必須同步到三個地方：

1. 本錯誤記憶庫。
2. `BNotes_MD圓桌會議_完成後自我檢視與回饋閉環SOP_20260517.md`。
3. 若可工具化，加入 validator 或 QA script。

## 九、MD 圓桌會議結論

BNotes 團隊接受以下標準：

錯誤不是失敗，重複錯誤才是系統失敗。  
返工不是浪費，沒有把返工轉成規則才是浪費。  
董事會指出問題不是打斷流程，而是幫團隊建立更高層級的判斷力。  

後續每一篇文章更新，都必須讓團隊比上一篇更準。若團隊不能從錯誤中建立不重犯能力，就不算真正完成文章升級。
