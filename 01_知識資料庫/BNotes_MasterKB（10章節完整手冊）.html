1專案總覽與核心理念

1.1 定位與目標

BNotes 焙學·原豆誌是一個以精品咖啡知識為核心的台灣自媒體部落格， 建立於 2024 年，採用「靜態優先、函數輔助」的混合架構， 同時滿足高效能 SEO 呈現與動態後台管理的需求。

面向規格
站點網域bnotescoffee.com
定位台灣精品咖啡知識媒體，SCA 標準 + 在地視角
內容量50+ 篇已發布文章，24+ 篇排程中
平均品質分90 分（自建評分系統 0–100）
分類數6 個主分類（手沖技法、義式咖啡、器材評測、產地風土、沖泡科學、咖啡生活）
Pillar Pages2 個（手沖咖啡完全指南、台灣咖啡產地風土）
Google Search Console45 次點擊 / 657 次曝光 / 6.8% CTR（建站 2 週內）

1.2 技術選型原則

本專案遵循以下三項核心原則選型，確保低成本、高效能且易於維護：

  1. 零基礎建設成本：使用 Cloudflare Pages 免費方案 + GitHub 免費倉庫，月費 0 元（除非流量超出免費額度）。
  2. 靜態優先、函數輔助：前台完全靜態（HTML/CSS/JS），CDN 邊緣節點快取。後台管理功能透過 Cloudflare Pages Functions（Serverless）實現，無需維護伺服器。
  3. Git 即資料庫：所有文章以 HTML 檔案儲存於 GitHub 倉庫，版本控制 = 備份系統，部署 = git push。
選型理由：為什麼不用 WordPress / Next.js / Hugo？
WordPress 需要 PHP 伺服器與資料庫，月費至少 NT$300+，攻擊面大。 Next.js 需要 Node.js 伺服器，SSR 成本高。Hugo/Jekyll 無法做動態後台。 本方案兼顧零成本、SEO 完整控制權與動態後台，是垂直媒體的最佳解。

// 文章狀態流轉

1. [生成]  AI 自動生成 / 人工撰寫

4.2 YAML Frontmatter 規格（完整）

所有排程文章的 HTML 檔案頂部必須包含 YAML frontmatter，在部署時由 strip-frontmatter.js 自動剝除。

---
title: "文章完整標題（SEO 最佳長度 30–70 字元）"
date: 2026-05-01                # 預定發布日，YYYY-MM-DD 格式（不加引號）
category: pour-over             # pour-over | espresso | equipment | terroir | science | lifestyle
description: "SEO meta description，70–160 字元，含目標關鍵字"
cover_image: "images/ai/{slug}-hero.jpg"
reading_time: 8                 # 預估分鐘數（整數）
focusKeyword: "主要 SEO 關鍵字"   # 用於品質掃描
coverAlt: "封面圖 alt 文字"      # 圖片無障礙 + SEO
status: draft                   # draft | approved（AI 文章審核流程使用）
---

<!-- 文章 HTML 正文從此開始 -->
關鍵注意：date 格式
date 欄位不可加引號，必須是裸值 2026-05-01。 加引號（"2026-05-01"）會導致 publish_scheduled.py 解析失敗， 文章永遠不會被自動發布（已修復：v2 版本加入 .strip('"').strip("'") 容錯處理）。

5後台管理系統

5.1 認證機制

後台位於 /admin/，所有請求經 functions/admin/_middleware.js 攔截驗證：

// 認證流程（HMAC-SHA256）
1. POST /admin/login { password }
   → deriveToken(password) = HMAC-SHA256(password, "bnotes-admin-gate-2026")
   → 設定 HttpOnly Cookie: bnotes_adm={token}
   → Max-Age: 8 小時, Secure, SameSite=Strict

2. 後續所有 /admin/* 請求
   → 讀取 Cookie → 驗證 HMAC token
   → 無效 → 302 redirect /admin/login
   → 有效 → 繼續處理

環境變數: ADMIN_PASSWORD (Cloudflare Pages Secret)
未設定時: 跳過認證（開發模式向下相容）

5.2 三 Tab 管理介面

TabAPI 來源功能
已發布GET /admin/articles-data列出 dist/articles/ 所有文章，可品質掃描、編輯、刪除、複製
排程文章GET /admin/scheduled-data列出 dist/_scheduled/ 文章，可調整日期、立即發布、品質修復、補圖

5.4 品質掃描評分系統

位於 functions/api/quality-scan.js，對文章 HTML 進行五維度自動評分：

維度滿分評估項目
A. SEO 完整性25 分title 長度（+3）、meta description（+5）、OG tags（+5）、Schema JSON-LD（+5）、canonical（+3）、frontmatter 完整（+4）
B. 內文質量30 分中文字數（+8，目標 2000+）、閱讀時間（+4，目標 6 分鐘+）、外部參考連結（+3）、內部連結（+3，需 2+ 篇）、作者/來源標示（+3）、深度分析關鍵詞（+2）、CTA 或延伸閱讀（+2）、AI 收錄適性（+5）
C. 圖片規範20 分Hero 圖片（+6）、OG:image（+4）、全部 img 有 alt（+6）、圖片使用 BNotes AI 圖庫（+4）
D. 結構可讀性15 分H1 唯一性（+3）、H2 數量 3+（+6）、段落/列表豐富性（+3）、表格/引用等專業元素（+3）
E. AI 知識收錄10 分結構化資料完整（+5）、深度分析詞彙（+5）
燈號：90+ 分為綠燈可立即發布；60–89 分為黃燈建議修正；60 分以下為紅燈需大幅修改。 目前全站平均：90 分，47/50 篇綠燈。

10.3 流量增長策略

基於 BNotes 實際案例（建站 2 週 45 次點擊 / 657 次曝光），整理出最有效的三個策略：

  1. Pillar Page 策略：每個核心主題建立一個 Pillar Page， 以 FAQPage Schema 抓取 Google Featured Snippet 流量。 目標：每個 Pillar Page 月流量 500-2000 次。
  2. Topic Cluster 內部連結：每篇子文章底部加入 3 條靜態內部連結， 指向 Pillar Page + 2 篇相關子文章，強化整個集群的 PageRank 傳遞。
  3. 每週定期更新：Google 對定期更新的站點給予更高爬取頻率。 利用 AI 管線每週生成 3 篇草稿，人工審核後排程發布， 維持「每週 2-3 篇新文章」的更新節奏。
當月流量穩定達到 10,000 PV 時，可考慮啟用：電子報訂閱（newsletter.html 解除隱藏）、 品牌合作頁面（collab.html 解除隱藏）、加入 Affiliate 連結（申請 Amazon Associates / 露天聯盟）。 達到 50,000 PV/月 可申請 Google AdSense；達到 100,000 PV/月 可申請 Mediavine 等高 RPM 廣告聯盟。
