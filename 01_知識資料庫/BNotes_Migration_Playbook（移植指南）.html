BNotes 架構 — 產業移植實戰手冊

從 BNotes 到任何產業
垂直部落格完整移植指南

本手冊假設你已取得 BNotes 倉庫，目標是在最短時間內 將整套架構（技術、設計、自動化）移植至你選定的新產業垂直領域， 並成功部署上線。

總工時約 6–8 小時 不需要改底層架構 最少 15 處品牌替換 D+1 可上線 適用任何知識型垂直媒體
依據：BNotes_MasterKB v1.0 倉庫：github.com/stanleytsoubiz/BNotes 文件版本：v1.0  ·  2026-04-21

移植前你需要確認的三件事

確認項目評估標準不符合怎麼辦
產業是否以文字為核心？以知識、觀點、教學為主要內容形態（非影片、商品、即時資訊）考慮改用有資料庫的架構（WordPress / Notion）
每月是否能產出 8–12 篇文章？自寫 2–4 篇 + AI 輔助 4–8 篇，維持每週 2 篇節奏縮減 AI 管線間距為兩週一次，降低品質門檻
目標讀者是否有搜尋習慣？核心主題在 Google Keyword Planner 月搜尋量 > 500先做關鍵字驗證再決定是否投入

1.2 定義 6 個分類（直接對應 BNotes 的 6 個分類）

BNotes 使用固定 6 個分類，移植時以相同數量做置換最省力（之後可調整）。為每個分類決定：英文 slug + 中文名稱。

命名規範
英文 slug 只能使用小寫英文字母、數字和連字號（a-z 0-9 -）， 不可使用中文、空格或底線。這個 slug 會出現在 URL、檔案名稱、程式碼中， 移植後幾乎不能再改，請謹慎決定。

Fork 倉庫 + 建立新專案
複製 BNotes 程式碼，建立獨立 GitHub 倉庫與 Cloudflare 專案
約 30 分鐘

2.1 Fork 或複製倉庫

有兩種方式，選一種即可：

方式 A：GitHub Fork（推薦）

  1. 前往 github.com/stanleytsoubiz/BNotes
  2. 點擊右上角「Fork」按鈕
  3. Repository name 改為新品牌名稱（例 VineNotes）
  4. 取消勾選「Copy the main branch only」（保留所有 branches）
  5. 點擊「Create fork」

方式 B：本地複製並推送至新倉庫

# 1. 複製 BNotes 倉庫
git clone https://github.com/stanleytsoubiz/BNotes.git VineNotes
cd VineNotes

# 2. 移除原來的遠端連線
git remote remove origin

# 3. 在 GitHub 建立新的空倉庫（請先在 github.com 手動建立），然後連接
git remote add origin https://github.com/你的帳號/VineNotes.git

# 4. 推送
git push -u origin main
bash

2.2 建立 Cloudflare Pages 專案

# 在本地倉庫根目錄執行
npx wrangler pages project create vinenotes
# 注意：此處的 vinenotes 是 CF 專案名稱，小寫，用連字號
# 執行後記下 account_id（顯示在 Dashboard 右側欄）
bash

2.3 更新 wrangler.toml

打開倉庫根目錄的 wrangler.toml，修改兩處：

# 修改前（BNotes 原設定）
name = "bnotes"
pages_build_output_dir = "./dist"

# 修改後（換成你的新專案名）
name = "vinenotes"        # ← 與 CF Pages 專案名稱一致
pages_build_output_dir = "./dist"  # ← 保持不變
wrangler.toml

2.4 更新 GitHub Actions 部署設定

打開 .github/workflows/deploy.yml，找到以下行並修改：

# 修改前
run: npx wrangler pages deploy --project-name=bnotes

# 修改後
run: npx wrangler pages deploy --project-name=vinenotes
.github/workflows/deploy.yml
此時先不要 git push
等 Step 3（品牌替換）完成後統一推送，避免 Cloudflare 部署失敗多次。

為什麼首批內容是關鍵？
Google 會在首次爬取時判斷站點的「內容深度」。 空站或少於 5 篇文章的站點難以被收錄。 建議上線時至少準備 10 篇 + 1 個 Pillar Page， 讓 Google 第一眼就判斷你是有深度的主題媒體。

內容節奏規劃

第 1–3 天

第 1–2 週
Google 收錄監控：每天到 Search Console 查看「覆蓋率」， 確認文章開始被收錄（通常 7–14 天開始出現）。若有「排除」錯誤立即修復。
第 2–4 週
AI 管線節奏：每週 AI 自動生成 1–2 篇草稿， 管理員後台審核後排程發布。人工每週至少補充 1 篇有個人觀點的深度文章。
第 30 天
首次復盤：查看 GSC 流量報告，找出「曝光多但點擊少」（CTR < 3%）的文章， 優化 title 和 meta description 提升 CTR。
