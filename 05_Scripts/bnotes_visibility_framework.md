## Part 2：SEO 升級清單（文章層級）

### 2.1 技術 SEO 檢查項目

每篇文章發布前，B8 與 B9 分工完成以下核查：

#### canonical URL
- 檢查項：頁面 `<link rel="canonical">` 是否指向該頁面本身的正式 URL
- 標準：HTTPS + 無 trailing slash 不一致 + 無參數污染
- 錯誤類型：Alternate canonical（目前 3 頁）→ 修復後重新提交 sitemap
- 負責：B9 修復，B8 確認

#### JSON-LD Schema（必填兩種）

**NewsArticle / BlogPosting（擇一，依文章性質）**
```
必填欄位：
- headline（= H1，≤ 110 字元）
- datePublished（ISO 8601）
- dateModified
- author.name
- publisher.name + publisher.logo
- image（至少 1 張，建議 1200×630）
- url（= canonical URL）
```

**FAQPage（有 geo-box 的文章必填）**
```
必填欄位：
- mainEntity 陣列
- 每個 Question.name = Q 原文
- 每個 Answer.text = A 原文（純文字，無 HTML）
```

- 檢查工具：Google Rich Results Test（https://search.google.com/test/rich-results）
- 負責：B9 植入，B8 驗證格式正確性

#### Open Graph 標籤
- 必填：`og:title` / `og:description` / `og:image`（1200×630）/ `og:url` / `og:type`
- `og:title` 可與 SEO title 相同；`og:description` 可與 meta description 相同
- 負責：B9 確認 Hugo 模板覆蓋正常

#### Breadcrumb Schema
- 所有分類頁與文章頁需有 BreadcrumbList JSON-LD
- 路徑示例：首頁 > 手沖咖啡 > 本文
- 負責：B9 模板層統一處理

#### Title 與 meta description 規範

| 欄位 | 長度規範 | 必含項目 |
|------|----------|----------|
| `<title>` | 30–60 字元 | 主關鍵字（前段）、品牌名（後段，用 ` | BNotes`）|
| meta description | 80–160 字元 | 主關鍵字、行動誘因或核心答案 |

- 超過 160 字元的 meta description：Google 截斷，不計分

### 2.2 內容 SEO 檢查項目

#### H1 唯一性
- 每頁僅一個 H1，內容 = `<title>` 的核心關鍵字段（允許小幅差異）
- 禁止：H1 與 H2 完全重複 / 頁面無 H1

#### 關鍵字自然密度
- 主關鍵字出現位置：H1、前 100 字、至少一個 H2、最後一段
- 密度目標：0.5%–2%（以正文字數計）
- 禁止：關鍵字堆砌（連續三個相同詞）

#### 內部連結
- 每篇文章必含 **2–3 個內部連結**，指向相關主題文章
- 錨文字使用描述性關鍵字，禁用「點這裡」「更多內容」
- 建議：B8 維護一份「熱門文章內部連結推薦表」，B3 寫作時參照

#### URL slug 規範
- 全英文小寫，連字符分隔（kebab-case）
- 不含日期、停用詞（a / the / of 等可省略）
- 長度建議 3–6 個詞
- 範例：`/pour-over-water-temperature`（佳）vs `/2025-12-25-pour-over-water-temperature-guide`（差）
- 已發布文章的 slug **不改動**（會破壞既有外部連結與索引）

## Part 3：GEO 升級清單（AI 引用層級）

### 3.1 geo-box Q&A 品質標準

每篇文章的 geo-box（FAQ 區塊）是 GEO 的核心載體。AI 引擎在回答問題時，優先引用「直接回答問題的獨立段落」。

#### 問題格式規範

- 使用完整問句，以問號結尾
- 問題應反映讀者實際搜尋行為（參考 GSC Search Queries）
- 每篇文章 geo-box：3–5 個 Q&A
- 問題類型覆蓋：定義型 / 方法型 / 比較型（建議每篇至少各一類）

範例：
- 定義型：「什麼是手沖咖啡的金杯萃取率？」
- 方法型：「手沖咖啡最適合的水溫是幾度？」
- 比較型：「淺焙和深焙咖啡豆的萃取時間有什麼差別？」

#### 答案長度與格式規範

| 項目 | 標準 | 說明 |
|------|------|------|
| 答案長度 | 40–80 字（繁中） | 太短 AI 跳過，太長被截斷 |
| 第一句 | 直接回答問句 | 不要以「關於這個問題...」開頭 |
| 數字/標準 | 必含具體數值（如有） | 「SCA 建議 90–96°C」比「偏高水溫」更易引用 |
| 來源標記 | 引用知名標準或研究 | 增加 AI 引用可信度 |
| 禁用語 | 「建議您...」「歡迎...」 | 服務話術降低引用率 |

#### 高引用觸發語言結構（GEO 寫作原則）

以下句型結構讓 AI 更容易引用 BNotes 內容：

1. **定義式**：「[術語] 是指 [定義]。依據 [來源]，標準為 [具體數值]。」
   - 範例：「手沖咖啡的金杯萃取率是指咖啡可溶物佔咖啡粉的比例。SCA 定義的金杯標準為萃取率 18–22%，濃度 1.15–1.55%。」

2. **量化比較式**：「相較於 [A]，[B] 的 [指標] 高出/低於 [數值]。」
   - 範例：「相較於 85°C，以 93°C 沖煮的手沖咖啡，酸質萃取效率提升約 15%，但苦味物質同步增加。」

3. **步驟指引式**：「[動作] 的標準步驟為：(1) … (2) … (3) …」
   - 結構清晰，AI 直接截取列表作為答案

4. **例外澄清式**：「常見誤解是 [A]，但正確做法是 [B]，原因在於 [機制]。」
   - 反常識結構天然吸引 AI 引用（同時提升社群貨幣性）

### 3.2 FAQPage Schema 植入流程

geo-box 與 FAQPage Schema 必須對應，執行步驟如下：

1. B3 撰寫文章時，geo-box 區塊使用指定 Hugo shortcode（`{{< geo-box >}}`）
2. B9 確認 Hugo 模板自動將 geo-box 內容轉為 FAQPage JSON-LD
3. B8 使用 Google Rich Results Test 驗證 Schema 輸出正確
4. 驗證通過後，提交該 URL 至 IndexNow（B9 執行）

**驗證標準：** Rich Results Test 顯示「FAQPage 找到 X 個問題」且無錯誤警告。

### 3.3 AI 引用追蹤方法（每月執行）

B8 每月月初進行一次 AI 引用基線追蹤，方法如下：

#### 測試關鍵字清單（月初更新）

選取當月 BNotes Top 10 高流量關鍵字，加上 5 個 GEO 目標關鍵字，共 15 個測試詞。

#### 測試平台與步驟

**ChatGPT（GPT-4o）**
1. 新開對話（清空 context）
2. 輸入：「[關鍵字問句]，請告訴我相關知識」
3. 記錄：是否提及 BNotes / bnotes.co / BNotes 的具體數據

**Perplexity**
1. 搜尋模式選「Web」
2. 輸入完整問句
3. 記錄：Sources 清單中是否出現 bnotes.co

**Google SGE / AI Overview**（如有）
1. Google 搜尋關鍵字
2. 截圖 AI Overview 區塊
3. 記錄：來源連結是否含 BNotes

#### 提升引用率的直接動作

若某關鍵字測試結果為「未引用」，B8 執行以下檢查：
1. 該關鍵字對應文章是否有 geo-box？（無 → 通知 B3 補充）
2. geo-box 答案是否符合 40–80 字直接回答格式？（不符 → B3 修改）
3. FAQPage Schema 是否正確生成？（錯誤 → B9 修復）
4. 文章是否已被 Google 索引？（未索引 → 優先處理索引問題）
