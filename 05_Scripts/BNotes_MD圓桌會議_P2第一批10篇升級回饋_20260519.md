## 本次學習

1. 不應把「長青知識文章」標成 NewsArticle。新聞型 Schema 只適合真正新聞或即時事件，BNotes 的核心知識庫文章應以 Article / BlogPosting 為主。
2. 導讀卡要使用「先把 X 讀成 Y」或等價語氣，讓讀者立刻進入 BNotes 的知識帶讀模式。
3. FAQ 必須優先使用折疊式 details，這符合目前已確認的閱讀順序：結語後接 FAQ，再接參考資料、分享、延伸閱讀。
4. 每次修改都必須同步檢查 source 與 dist，否則下一次發布可能把舊錯誤帶回正式站。
5. P2 不應一次混做所有文章深修；先用 10 篇一批降低錯誤率，完成後回到 MD 圓桌會議學習，再開下一批。

## 抽查後追加修正

董事長抽查後提出三點有效回饋，已列入後續 QC：

1. AeroPress 不可出現重複主標。修正方式：移除內文舊 article-header 主標，保留統一文章標題；並補入「三種常用杯型」表格，讓內容更符合完整指南定位。
2. 精品咖啡 vs 商業咖啡不可只靠隱藏 hero 或舊外殼呈現主標。修正方式：補入標準 bnotes-header 與可見 H1，移除舊 hero 主標殘留。
3. 參考資料需維持一致樣式。修正方式：磨豆機清潔文章參考資料由 ul 改為 ol，與其他標準文章一致。

後續每批升級後，抽查至少要包含：可見主標唯一性、導讀卡、FAQ 折疊、參考資料列表型式、分享區、延伸閱讀與手機閱讀觀感。

## 後續防錯查核重點

MD 委員會與專家團隊確認：未來避免類似錯誤，不能只靠「下次注意」，必須轉成機器查核、PGM 驗收與人工抽驗三層閉環。

1. 主標唯一性列為 P1：每篇文章正式輸出後只能有 1 個靜態 H1。若出現舊 hero、隱藏主標、內文重複主標或腳本注入造成混亂，必須先修正再進入下一篇。
2. 標準文章外殼列為 PGM 驗收：可見標題、Logo、品牌副標、文章寬度與分享區需對齊已確認標竿文章，不可只用局部內文判斷是否完成。
3. 參考資料格式列為 P1：參考資料區需使用一致列表規格，避免同批文章出現 ol / ul 混用造成字體、間距與視覺秩序不一致。
4. 完整指南需做內容深度確認：若標題承諾「完整指南」，內文必須包含可操作的判斷表、對照矩陣、常見情境或排錯邏輯，不能只有概念型段落。
5. 每次修改必須 source / dist 雙檢：修 source 後要同步正式輸出；修 dist 後要回補 source，避免下次部署覆蓋。
6. 每批部署後必須跑自動 QC，再做人工抽驗：自動 QC 負責抓結構錯誤，人工抽驗負責閱讀舒適度、語氣、圖卡、分享區與手機觀感。
7. 錯誤進入學習庫：每次董事長或團隊抽查發現的錯誤，都要轉成「可查核句子」，例如「參考資料不得使用 ul」或「導讀卡三點間距不得明顯大於標竿文章」，而不是只留下籠統提醒。

本次已將「H1 主標唯一性」、「隱藏 hero 不可作為主標完成依據」與「參考資料列表規格」加入 `05_Scripts/bnotes-deployment-qc.js`，之後每次批次更新都會被自動掃描。新增規則第一次掃描後，目前抓出 36 個 P1，這些會進入下一階段的修正清單，而不是視為已通過。

## 2026-05-20 MD 圓桌會議追蹤確認

本輪依照新增 QC 門檻持續推進，優先清理 P1 層級的品牌一致性問題。MD 委員會與專家團隊確認：這類問題會直接影響讀者對 BNotes 版型穩定度與專業性的感受，因此應先處理，再進入下一批內容深修。

### 本輪處理

1. 將重複 H1 的文章改為只保留一個可見主標，舊 hero 中的標題改為非 H1，避免 SEO 與閱讀語意混亂。
2. 將參考資料區統一為 ol 列表，讓字體、縮排與閱讀秩序對齊標竿文章。
3. 修正 QC 誤判：參考資料區只檢查 references block 本身，不再把後方延伸閱讀的 ul 誤判為參考資料錯誤。

### 驗收結果

- P0：0
- P1：36 -> 0

### 本次學習

1. 自動 QC 也需要被審查。若 QC 範圍過寬，會造成誤判；因此 PGM 要確認每條規則是否準確對應真正問題。
2. 主標問題不能只看「頁面是否有 H1」，要確認讀者看見的是正確標題，且不被舊 hero 或舊模板干擾。
3. 參考資料與延伸閱讀雖然都在文章尾端，但語意不同，未來 QC 必須分區判斷，避免混淆。
